# 依頼分析OAシステム ソリューション提案

## 作成日: 2026年4月30日（初版）
## 作成者: データ解析ソリューション課（市川）

---

## 0. 要約（先に結論）

- 現相談の中核は **「48,512件の死蔵データを自然言語で再活用できる検索基盤」** の構築。本質はRAG（検索拡張生成）の社内導入。
- 技術スタックは **「ハイブリッド検索（ベクトル＋全文）＋ LLMによるメタデータ事前抽出 ＋ 検索結果のLLM要約」** の3層構成を推奨。
- 業務プロセス改善（依頼受付の自動化）は ServiceNow 計画に委ね、本テーマは **検索・再活用基盤** に集中する。
- ご提案の **「Claude Code を用いたLLM Wiki／インデックス化」** は方向性として正しい。ただし「Claude Code（CLIツール）を業務担当者に直接使ってもらう」のは現実的ではない。**位置づけを整理した上で採用する**（後述 §5）。
- セキュリティ制約（顧客情報／輸出管理データの混在可能性）から、**社内Azure OpenAI＋社内クローズド環境** を前提とした設計を強く推奨。

## 1. 解くべき問題の本質

| 表層の課題 | 本質 | 解の方向 |
|------------|------|----------|
| 過去データが検索できない | 検索キーが担当者の任意設定で属人化 | **検索キーを人手設定に依存しない**（本文を機械的に意味検索） |
| 受付Noしか返らない | 結果の要約がない | **LLMによる要約をインデックス時に事前付与** |
| 画像PDFが混じっている | OCR前提のため検索インデックスに乗せられない | **OCR込みのインデックス構築パイプライン** |
| 担当者ごとに引き当て力が違う | 暗黙知への依存 | **検索基盤＋要約UIで知識を可視化・民主化** |

## 2. 推奨アーキテクチャ（3層構成）

```
┌─────────────────────────────────────────────────────────────┐
│  ① インジェスト層（バッチ：初期48,512件＋差分）              │
│   ファイル取得 → テキスト抽出 → OCR → 正規化                  │
│   → LLMでメタデータ抽出（依頼内容／分析手法／主要結果／顧客名匿名化） │
│   → チャンク分割（章単位＋スライディング）                     │
└──────────────────────────┬──────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  ② インデックス層（社内クローズド環境）                       │
│   ベクトルDB（埋め込み） ＋ 全文検索（BM25）＋ メタデータDB    │
│   ハイブリッド検索 ＋ リランカー                              │
└──────────────────────────┬──────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  ③ 利用層（業務担当者向けUI）                                 │
│   Web画面：自然言語入力 or 依頼書ファイルアップロード         │
│   結果カード：受付No／類似度／分析手法要約／結果要約／原本リンク │
│   LLMによるオンザフライ追加要約                               │
└─────────────────────────────────────────────────────────────┘
```

### 各層の技術選定候補

| 層 | 候補A（社内環境優先） | 候補B（Microsoft親和性） | 候補C（軽量PoC） |
|----|----------------------|--------------------------|-------------------|
| ①OCR | Azure Document Intelligence | Azure AI Document Intelligence | Tesseract + 前処理スクリプト |
| ①メタデータ抽出 | 社内Azure OpenAI（GPT-4系） | Azure OpenAI | Anthropic API（Claude） |
| ②ベクトルDB | Azure AI Search（ベクトル機能）| Azure AI Search | FAISS / Qdrant（社内サーバ） |
| ②全文検索 | Azure AI Search（BM25併用） | 同左 | Elasticsearch / Meilisearch |
| ③UI | 内製Webアプリ（Flask 等） | Copilot Studio + Power Apps | Streamlit（PoC段階） |

- 機密／輸出管理／顧客名秘匿対象の整理（C-07）

## 4. 技術的キーポイント

### 4.1 「分析手法・結果の要約」の作り方

- **インデックス時の事前抽出**を主とし、検索時のオンザフライ生成は補助的に。
- 報告書から「依頼内容」「使用した分析手法（装置・条件）」「主要結果（結論3行）」を構造化抽出。
- プロンプト例：
  ```
  以下は分析報告書の本文です。次の3項目をJSONで抽出してください。
  - request: 依頼者の依頼内容（1〜2文）
  - method: 使用した分析手法・装置・主要条件（箇条書き）
  - result: 主要結果と結論（3行以内）
  抽出できない項目は null としてください。
  ```
- 抽出結果はメタデータDBに保存し、検索結果カードに直接表示。

### 4.2 ハイブリッド検索の必要性

- ベクトル検索のみ：意味的類似は強いが、製品型番・装置名など固有表現に弱い
- 全文検索のみ：固有表現に強いが、表現ゆれに弱い（既存検索の問題そのもの）
- **両者をスコア統合**することで、本テーマの「表記ゆれ＋専門用語」を両立カバー

### 4.3 OCRの現実的な落としどころ

- まずテキスト層を持つPDFをそのまま処理し、テキスト層のないPDFのみOCR
- 化学式・装置型番は誤認識が多発しがちなので、**OCR後のテキストはあくまで検索インデックス用**と割り切り、原本リンクで人が確認する設計
- 古い手書き帳票がある場合は対象外（人手レビューフラグ）

### 4.4 セキュリティ設計

- 全工程を社内ネットワーク内に閉じる前提
- 社内Azure OpenAI（プライベートエンドポイント）を利用
- 顧客名・案件名のマスキングを検討（メタデータDBには素のまま保持し、外部露出経路でマスク）
- 機密区分のメタデータタグを必ず付与し、利用者の権限と突き合わせて検索結果フィルタ

## 5. ご提案いただいた「Claude Code でのLLM Wiki／インデックス化」案への評価

### 結論

**「Claude Code（CLIツール）を業務担当者の検索UIにする」のは推奨しない**が、**「Claude Code／Claude Agent SDK／Anthropic API のエージェント機能を、インデックス構築・メタデータ抽出のバッチ処理に活用する」のは強く有効**です。位置づけを切り分けた上での採用を提案します。

### 5.1 良い点（採用すべき側面）

| 観点 | 良い点 |
|------|--------|
| メタデータ抽出の品質 | Claudeは長文の構造化抽出に強く、報告書の章立てが揺らいでも安定して抽出できる |
| ファイル横断処理 | Agent SDK ＋ MCP で、社内ファイルサーバ上のWord／Excel／PPT／PDFをツール越しに直接処理できる |
| 反復改善 | プロンプトを Skill として定義しておけば、チューニングや業務知識の追加が容易 |
| 開発生産性 | Claude Code 自体を「PoC段階の自動化スクリプトを書く道具」として開発者が使える（業務担当者に渡す道具ではない） |
| Wiki化の発想 | 各案件ごとに「依頼／手法／結果／類似案件リンク」を持つ Markdown を生成して社内Wikiに置く構成は、検索基盤に加えて**人が直接読める二次情報**として価値が高い |

### 5.2 注意すべき点（このまま採用しない方がよい側面）

| 観点 | 懸念 | 対処 |
|------|------|------|
| 業務担当者のUI | Claude Code はエンジニア向けCLIツール。担当者・新人が日常的に使う検索UIにはならない | 検索UIは別途Web画面として用意（§2 ③） |
| データ送信先 | デフォルトで Anthropic API に送信される。社内機密データの取り扱いには不適 | Azure経由のClaude（Anthropic on AWS／Azure経由）か、社内Azure OpenAI（GPT系）を採用 |
| 課金・予算 | 48,512件 ×（依頼書＋報告書）の初期処理はトークン消費が大きい | フェーズ1のPoCで単価を実測し、本番見積りに反映 |
| 監査・運用 | エージェント実行のログ／再現性／障害時の復旧設計が独立に必要 | バッチ処理基盤として正式に位置づけ、ジョブ管理／ログ管理を整備 |
| ベンダーロック | Claude固有のSkill／MCPに最適化しすぎると将来移植性が下がる | プロンプトと抽出スキーマは LLM非依存の形で管理し、実行ランタイムだけ差し替え可能にする |

### 5.3 推奨される位置づけ

```
[業務担当者]                    [PoC開発・運用]                 [LLM]
   │                                │                           │
   ▼                                ▼                           ▼
Web検索UI ◀── 検索API ── インデックスDB ◀── 抽出パイプライン ── 社内承認済みLLM
                                       │              ▲
                                       │              │
                                       └─ Claude Code/Agent SDK でこのパイプラインを開発・運用
                                       （Skillsとしてプロンプトと処理を体系化）
```

- **担当者が触るのはWeb検索UI**（自然言語入力＋ファイルアップロード）
- **Claude Code は「裏側のパイプラインを作る・運用する道具」** として開発者・運用者が使う
- **LLM Wiki 化（各案件のMarkdownカード生成）** はインデックス化と並行して有用。社内SharePoint／GitLab Wiki にも展開可能

### 5.4 「LLM Wiki」のもう一段の解釈

ご提案の "LLM Wiki" には以下の解釈が考えられます。それぞれ価値が異なります。

| 解釈 | 内容 | 評価 |
|------|------|------|
| (a) 各案件の自動生成サマリーをWiki化 | 1案件＝1ページのMarkdown。タグ・分析手法・装置・結果を構造化 | **強く推奨**。検索基盤と相補的。人が直接ブラウズ可能 |
| (b) 横断ナレッジ（手法別／装置別の集約ページ） | 「SEM観察での過去事例まとめ」「特定故障モード事例集」 | **PoCで価値検証する候補**。新人教育に直結 |
| (c) Claude Code を業務担当者のUIにする | CLIから検索 | **非推奨**。業務利用には不適 |
| (d) Anthropic API を組み込んだ社内Webアプリ | 自然言語検索＋要約のフロント | **推奨**。ただしAPIキーやデータ送信先のガバナンス必要 |

### 5.5 「LLM Wiki」と「RAG検索基盤」の関係についての補足

両者は **対立概念ではなく、同じRAG基盤の上のビューの違い** として整理するのが正確である。

```
            [Azure AI Search + Azure SQL]  ← 1つの真実の源
                       │
        ┌──────────────┼─────────────────┐
        ▼                                 ▼
[検索UI（FastAPI+React）]      [Wikiビュー（自動生成Markdown）]
 主用途: 依頼書ファイル検索      主用途: 新人教育・体系的閲覧
         類似案件の即時提示             横断ナレッジページ
```

| 観点 | 検索UI中心 | Wiki中心 | 両立構成 |
|------|----------|---------|---------|
| ユーザーの第一エントリ | 検索ボックス | 階層ブラウズ＋検索 | 両方 |
| 1案件あたりのページ生成 | 不要 | 必要 | 必要 |
| URL の永続性 | 検索結果URL | 案件ごとに固定URL | 両方OK |
| 人手編集（補足追記） | 不可 | 可能 | Wiki側で可 |
| 生成・運用コスト | 低 | 高 | 高 |
| 新人の体系的学習 | 弱い | 強い | 強い |
| 依頼書ファイルでの類似検索 | ◎ | バックエンド次第 | ◎ |

**結論**: Wikiは「検索ができない」のではなく「検索基盤を意識的に設計する必要がある」。後ろに同じRAG基盤を共有する前提で、検索UIとWikiビューの **両立構成** を採るのが本テーマでの最適解。コスト・運用負荷とのトレードオフでフェーズ判断（PoCは検索UIのみ、本番でWiki追加など）。
