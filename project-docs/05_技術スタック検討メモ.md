## 0. 前提

- 原本ファイル（48,512件のPDF/Word/Excel/PPT）はOAサーバに据え置き、移動・複製しない
- 顧客情報／輸出管理データの混在可能性があるため、**全コンポーネントを社内ネットワーク内**に閉じる
- 既存M365／Azure環境との親和性を優先（社内Azure OpenAI が使える前提）
- ServiceNow計画とは疎結合で、検索基盤は単独で動作可能

## 2. コンポーネント別の技術選定

### 2.1 インジェスト層

| 工程 | 採用技術 | 代替案 | コメント |
|------|---------|--------|----------|
| ファイル取得 | Azure Container Apps Job + SMB マウント | Azure Storage 経由 | 原本据え置き前提なら SMB 直読み |
| PDF抽出（テキスト層あり） | `pdfplumber` または `pypdf` | Apache Tika | 高速・無料 |
| OCR（画像PDF） | **Azure AI Document Intelligence (Layout / prebuilt-document)** | Tesseract + 前処理 | 表構造・章立て認識が強い／日本語精度も実用レベル |
| Word | `python-docx` | Apache POI | 標準的 |
| Excel | `openpyxl` + シート単位文章化 | pandas | 表データはLLMで意味化要 |
| PowerPoint | `python-pptx` | Apache POI | スライド単位でチャンク |
| メタデータ抽出 | **Azure OpenAI GPT-4o**（プライベートエンドポイント） | GPT-4o-mini（コスト圧縮） | 構造化抽出は4o系が安定 |
| チャンク分割 | LangChain RecursiveCharacterTextSplitter / 自前実装 | LlamaIndex SentenceSplitter | 1000-1500トークン、200オーバーラップ |
| 埋め込み生成 | **Azure OpenAI text-embedding-3-large**（3072次元） | multilingual-e5-large（OSS） | 日本語専門用語に強い／クロス言語対応 |
| パイプライン制御 | Azure Container Apps Job + Azure Service Bus | Apache Airflow | 差分実行・再試行・進捗管理 |

### 2.2 インデックス層

| コンポーネント | 採用技術 | 代替案 | コメント |
|---------------|---------|--------|----------|
| 検索エンジン | **Azure AI Search (Standard S2)** | OpenSearch + Qdrant 自前構築 | ベクトル＋BM25＋semantic rankerが1製品で完結 |
| メタデータDB | **Azure SQL Database (Basic/Standard)** | PostgreSQL on Azure | 既存M365親和性・運用負担小 |
| ファイルストレージ | OAサーバ据え置き（原本）／Azure Blob（抽出テキスト・OCR結果のキャッシュ） | - | 原本は動かさない |

### 2.3 検索・利用層

| コンポーネント | 採用技術 | 代替案 | コメント |
|---------------|---------|--------|----------|
| 検索API | **Azure Functions (Python)** または Container Apps | App Service | レスポンス10秒以内目標 |
| Web UI | **FastAPI + React + Azure AD認証** | Streamlit (PoC段階のみ) | 業務向けはFastAPI推奨 |
| 認証・認可 | **Microsoft Entra ID (旧Azure AD)** | - | 社内SSO前提 |
| 監査ログ | Azure Monitor + Application Insights | - | 利用者・クエリ・結果のログ取得 |

### 2.4 セキュリティ・運用

| 観点 | 採用技術 |
|------|---------|
| ネットワーク分離 | VNet + Private Endpoint（Azure AI Search／OpenAI／SQL すべて） |
| 暗号化 | 通信TLS、保存時はAzure既定の暗号化＋必要に応じてCMK |
| 機密区分 | メタデータDBに `confidentiality_level` を必須付与、検索APIで利用者ロールと突合 |
| データ保持 | Azure OpenAI 「データ保持なし」設定（Abuse Monitoring opt-out要申請） |
| バックアップ | Azure SQL の自動バックアップ／Azure AI Search の手動再インデックス手順 |
| 監査 | 検索ログ・LLM入出力ログ（機密マスク後）を保管 |

### 3.3 検索クエリの仕組み（ハイブリッド検索）

```
[ユーザー入力] (自然言語 or 依頼書ファイル)
    │
    ├─ ファイルの場合: LLMで「依頼意図」を抽出してクエリ化
    │
    ▼
[クエリ正規化]
    │
    ├─ ベクトル検索 (chunk_vector に対して)         ─┐
    ├─ キーワード検索 (BM25, chunk_text)            ─┼─ Reciprocal Rank Fusion で統合
    └─ semantic ranker (Azure AI Search ネイティブ) ─┘
    │
    ▼
[chunk → case 集約] (case_id でグループ化、最大スコアを採用)
    │
    ▼
[フィルタ適用] (機密区分・拠点・年範囲)
    │
    ▼
[結果カード生成]
   - 受付No
   - 案件タイトル
   - 拠点・年
   - request_summary / method_summary / result_summary （メタデータDBから引く）
   - 一致した代表チャンク（ハイライト）
   - 原本ファイルへのリンク
```

## 4. インジェスト処理の具体的な擬似コード

```python
# 1ファイルあたりの処理（Azure Container Apps Job として並列実行）

def process_case(case_id: str, file_paths: list[str]):
    chunks_all = []
    full_text_all = []

    for path in file_paths:
        ext = path.split('.')[-1].lower()

        if ext == 'pdf':
            text = extract_pdf_text(path)
            if not text or text_is_too_short(text):
                # 画像PDFと判断 → Azure Document Intelligence
                text = azure_doc_intelligence_ocr(path)
                ocr_used = True
            else:
                ocr_used = False
        elif ext == 'docx':
            text = extract_docx(path)
        elif ext == 'xlsx':
            text = extract_xlsx_as_text(path)  # 表は意味化
        elif ext == 'pptx':
            text = extract_pptx(path)
        else:
            continue

        full_text_all.append(text)

        # チャンク分割（章境界を尊重）
        for chunk in split_into_chunks(text, max_tokens=1500, overlap=200):
            chunks_all.append({
                'case_id': case_id,
                'file_path': path,
                'file_type': ext,
                'chunk_text': chunk.text,
                'section': chunk.section,
                'ocr_used': ocr_used,
            })

    # メタデータ抽出（ファイル横断、要約はLLMが全文を読んで生成）
    metadata = extract_metadata_with_llm('\n'.join(full_text_all))
    # metadata = {
    #   'title': ..., 'request_summary': ..., 'method_summary': ...,
    #   'result_summary': ..., 'instruments': [...], 'materials': [...],
    #   'confidentiality_level': ...
    # }

    # 埋め込み生成（バッチで効率化）
    vectors = azure_openai_embed([c['chunk_text'] for c in chunks_all])
    for c, v in zip(chunks_all, vectors):
        c['chunk_vector'] = v

    # 書き込み
    upsert_metadata_db(case_id, metadata, file_paths)
    upsert_search_index(chunks_all)
```

## 5. サイジングとコスト概算（オーダー感）

### 5.1 ボリューム想定

| 項目 | 値 | 根拠 |
|------|-----|------|
| 案件数 | 48,512件 | PPTX記載 |
| 1案件あたりファイル数 | 約3〜5（依頼書＋報告書＋データ） | 想定 |
| 1案件あたり総ページ数 | 平均30ページ | 想定（要Q-12確認） |
| 1案件あたりトークン数 | 約15,000トークン | 1ページ500トークン目安 |
| 総トークン数 | **約7.3億トークン** | |
| 画像PDF比率 | 30〜50%（仮置き） | 要Q-02確認 |
| 総OCRページ数 | 約20〜35万ページ | |
| 総チャンク数 | **約50万チャンク** | 1500トークン/チャンク |
| ベクトルストレージ | 約6GB | 3072次元 × 4byte × 50万 |

### 5.2 初期インデックス構築コスト（一回限り）

| 項目 | 試算 | コメント |
|------|------|----------|
| OCR (Azure Document Intelligence) | 20万ページ × 約 ¥2/ページ = **約40万円** | Layout モデル想定 |
| 埋め込み (text-embedding-3-large) | 7.3億トークン × $0.13/1Mt ≒ $95 = **約1.5万円** | 1回処理 |
| メタデータ抽出 (GPT-4o) | 7.3億トークン入力 × $5/1Mt + 出力少 ≒ $3,650 = **約55万円** | コスト最大要素 |
| メタデータ抽出（GPT-4o-miniの場合） | 7.3億トークン入力 × $0.15/1Mt ≒ $110 = **約1.7万円** | 精度を見て選定 |
| **初期構築合計** | **GPT-4o使用時: 約100万円 / mini使用時: 約45万円** | 一回限り |

### 5.3 月次運用コスト（PoC〜小規模本番）

| 項目 | 試算 | コメント |
|------|------|----------|
| Azure AI Search (Standard S2) | **約10万円/月** | 50万chunkは S1 で足りるが S2 推奨 |
| Azure SQL Database (Standard S2) | **約3万円/月** | |
| Azure OpenAI (検索時要約) | 月1,000検索 × 平均5万トークン × $5/1Mt ≒ $250 = **約4万円/月** | 利用量次第 |
| Azure Container Apps + Functions | **約2万円/月** | |
| Document Intelligence (差分OCR) | **数千円/月** | 新規案件のみ |
| **月次合計** | **約20万円/月** | |

### 5.4 コスト削減の余地

- メタデータ抽出を GPT-4o-mini にすると初期構築が **1/10〜1/30** に圧縮可能 → 抽出精度の検証必須
- 埋め込みを `text-embedding-3-small` (1536次元) にすると **1/4** に圧縮可能 → 検索精度の検証必須
- Azure AI Search を Basic にすると約4万円/月だが、ベクトル容量制約あり → 全件本番では S1〜S2 必須

## 7. ServiceNow計画との連携設計

| 連携パターン | 説明 |
|-------------|------|
| **(a) 疎結合 / リンクのみ** | ServiceNow チケットから「過去事例検索」リンクで本Webアプリへ遷移。初期推奨 |
| **(b) 中結合 / API呼び出し** | ServiceNow フォーム入力時にバックグラウンドで検索APIを呼び、候補を表示 |
| **(c) 密結合 / iframe組込** | ServiceNow 画面内に検索UIを iframe 埋込 |

PoC段階は **(a)** で十分。本番化時に **(b)** へ拡張するのが現実的な進め方。

## 8. 採用しなかった選択肢とその理由

| 候補 | 不採用理由 |
|------|-----------|
| **Anthropic API 直結** | 社外送信のためデータガバナンス上ハードル高い。社内Azure OpenAIの方が承認が通りやすい |
| **オンプレOSS（Qdrant + Tesseract + Llama）** | 構築・運用負担が大きく、人員リソースに見合わない。日本語OCR・LLMの精度も劣る |
| **Copilot Studio 単体構成** | 48,512件はProjects型ナレッジには載らない／構造化検索・メタデータ運用が弱い |
| **Claude for Work 単体本番運用** | 同上。PoC段階なら有用（[03_ソリューション提案.md §5](03_ソリューション提案.md) 参照） |
| **検索エンジンを ElasticSearch 単体** | ベクトル検索を別途構築する手間。Azure AI Search はハイブリッドが標準機能 |
