## 3. 課題詳細

### C-02: 過去48,512件のデータ資産が死蔵【本テーマの中核】

| 項目 | 内容 |
|------|------|
| **背景・現状** | 既存OAシステムにキーワード検索機能はあるが、キーワードは各担当者が任意設定する仕様。表記ゆれ・粒度ばらつき・属人化により、第三者が過去事例を引き当てるのは困難 |
| **目的（ゴール）** | 自然言語または依頼書ファイルそのものを入力として、48,512件を横断的に検索し、類似案件の分析手法・結果を提示できる仕組みを構築する |
| **業務価値** | (1) 新規案件の実験計画立案を高速化 (2) 重複依頼の検出 (3) 技術伝承・新人教育の効率化 (4) 蓄積データ資産の投資対効果回収 |
| **受入条件（仮）** | (1) 既存依頼書（Word／PDF）をそのまま入力できる (2) 受付No＋分析手法＋結果要約のセットで上位N件を返す (3) 画像PDF・Word・Excel・PPTが検索対象 |
| **技術的アプローチ（案）** | RAG（ベクトル検索＋全文検索のハイブリッド）／メタデータ自動抽出／LLM要約。詳細は `03_ソリューション提案.md` 参照 |
| **必要データ** | 過去依頼書PDF、報告書 Word／Excel／PPT、受付Noなどのメタデータ |
| **リスク・懸念** | (1) 画像PDFのOCR精度 (2) 機密情報の社外LLM送信不可 (3) 表記ゆれ・専門用語の同義語処理 (4) 48,512件の初期インデックス構築コスト |

---

### C-03: 画像PDFを含む混在ファイルのOCR

| 項目 | 内容 |
|------|------|
| **背景・現状** | 古い依頼書は画像PDFの比率が高いと推定（要ヒアリング）。報告書は Word／Excel／PPT が混在 |
| **目的（ゴール）** | すべてのファイル種別から本文テキストを抽出し、検索可能な状態に正規化する |
| **技術的アプローチ（案）** | (1) Word／Excel／PPT は標準ライブラリで抽出 (2) PDF はテキスト層 → なければOCR (3) OCRは Azure Document Intelligence／Tesseract／Apache Tika 等を比較 (4) 画質劣化が大きい古い案件は人手レビューフラグ |
| **リスク・懸念** | OCR誤認識率が高いと検索性に直結。化学式・製品型番など専門表記の誤認識リスク |

### C-04: 検索結果への「分析手法・結果の要約」付与

| 項目 | 内容 |
|------|------|
| **背景・現状** | 既存検索は受付Noの羅列のみ。利用者が報告書を1件ずつ開いて確認する手間が残る |
| **目的（ゴール）** | 検索結果に各案件の「依頼内容／使用した分析手法／主要結果」の要約を自動付与し、一覧で比較検討可能にする |
| **技術的アプローチ（案）** | (1) インデックス構築時にLLMでメタデータ抽出（バッチ） (2) 検索時にトップN件についてオンザフライで要約再生成（必要に応じ） (3) プロンプトテンプレートを案件種別ごとに最適化 |
| **リスク・懸念** | 要約のハルシネーション、専門用語の誤訳、原本との乖離。出典リンク必須 |

## 5. 優先度決定の観点

1. **データ資産活用の中核（C-02・C-04）が最優先**: 現行OAシステムの最大の不満点であり、PoCのコア機能
2. **OCR（C-03）はC-02の前提条件**: 画像PDF比率の確認結果次第で工数が大きく変動
3. **セキュリティ（C-07）は早期にクリアすべき**: 構成選定のすべてに先行する制約条件
4. **業務プロセス改善（C-01）はServiceNow計画と整合**: 検索基盤と切り離して並行進行が可能
5. **評価指標（C-10）はPoC計画と同時に設計**: 後付けでは効果が示せない
